# מדריך הקמת Supabase + פרסום ב-GitHub Pages

מדריך זה מסביר צעד-אחר-צעד איך לחבר את אתר "קפסולת הזמן" ל-Supabase (לתיעוד מי
לחץ על קישור התשלום) ואיך לפרסם אותו ב-GitHub Pages. אין צורך בידע קודם.

---

## חלק א' — הקמת Supabase

### 1. יצירת פרויקט
1. היכנסו ל-<https://supabase.com> והתחברו (אפשר עם GitHub).
2. לחצו **New project**.
3. בחרו ארגון, תנו שם (למשל `time-capsule`), בחרו סיסמה ל-DB (שמרו אותה בצד),
   ובחרו Region קרוב (למשל `Frankfurt (eu-central-1)`).
4. לחצו **Create new project** והמתינו ~2 דקות שהפרויקט יוקם.

### 2. יצירת הטבלה והרשאות (RLS)
1. בתפריט הצד בחרו **SQL Editor** → **New query**.
2. העתיקו את כל התוכן של הקובץ [`supabase/schema.sql`](./supabase/schema.sql)
   והדביקו בעורך.
3. לחצו **Run**. אמורה להופיע הודעת הצלחה.

מה זה עשה:
- יצר טבלה בשם `ticket_clicks` שמאחסנת: שם מלא, גיל, מספר כרטיסים, עם מי מגיעים,
  המחיר שחושב, וכן `user_agent` ו-`referrer` (מאיזו פרסומת הגיעו).
- הפעיל **Row Level Security** והגדיר מדיניות שמאפשרת **רק הוספה (INSERT)**
  מהדפדפן (מפתח ה-anon הציבורי). הדפדפן **לא יכול לקרוא, לשנות או למחוק** רשומות
  של אחרים — כך הנתונים מוגנים. אתם, כבעלי הפרויקט, רואים הכל דרך הדשבורד.

### 3. השגת המפתחות
1. בתפריט הצד: **Project Settings** (גלגל שיניים) → **API**.
2. העתיקו שני ערכים:
   - **Project URL** — למשל `https://abcd1234.supabase.co`
   - **anon public** key (תחת "Project API keys") — מחרוזת ארוכה.

> ⚠️ **חשוב:** השתמשו אך ורק במפתח **anon public**. לעולם אל תעתיקו את מפתח
> ה-`service_role` לאתר/ל-GitHub — הוא נותן גישה מלאה לכל הנתונים.

### 4. צפייה בנתונים שנאספו
בתפריט הצד: **Table Editor** → בחרו את הטבלה `ticket_clicks`. כל לחיצה על
"מעבר לתשלום" באתר תיצור כאן שורה חדשה. אפשר לייצא ל-CSV דרך הכפתור **Export**.

---

## חלק ב' — קישור שירות התשלום

קבלו מההנהלה את כתובת ה-URL של שירות הסליקה (העמוד שאליו הגולש מועבר כדי לשלם).
נשתמש בה כמשתנה בשם `VITE_PAYMENT_URL`.

> הזרימה באתר: הגולש ממלא שם → גיל → מספר כרטיסים → עם מי מגיע. בלחיצה על "מעבר
> לתשלום", **קודם** נשמרים הפרטים ב-Supabase, **ואז** הוא מועבר לכתובת התשלום.
> הנתונים נאספים לפני ההפניה כי ייתכן שלא נוכל להחזיר את הגולש אלינו אחרי התשלום.

---

## חלק ג' — פרסום ב-GitHub Pages

### 1. העלאת הקוד ל-GitHub
האתר מוכן לכך. צרו ריפו חדש ב-GitHub ודחפו את התיקייה הזו (ראו פקודות בתחתית
`IMPLEMENTATION.md`). ה-branch הראשי צריך להיקרא `main`.

### 2. הגדרת המשתנים בריפו
ב-GitHub: **Settings** → **Secrets and variables** → **Actions** → לשונית
**Variables** → **New repository variable**. הוסיפו שלושה משתנים:

| Name (שם) | Value (ערך) |
|---|---|
| `VITE_SUPABASE_URL` | ה-Project URL מסעיף א'.3 |
| `VITE_SUPABASE_ANON_KEY` | מפתח ה-anon public מסעיף א'.3 |
| `VITE_PAYMENT_URL` | כתובת שירות התשלום מחלק ב' |

> שלושת הערכים האלה ציבוריים מעצם טבעם (הם מגיעים לדפדפן), ולכן "Variables" מתאים.
> גם "Secrets" יעבוד. הטבלה מוגנת ע"י RLS, לא ע"י הסתרת המפתח.

### 3. הפעלת GitHub Pages
ב-GitHub: **Settings** → **Pages** → תחת **Build and deployment** → **Source**
בחרו **GitHub Actions**.

### 4. פרסום
כל דחיפה (push) ל-branch `main` מפעילה אוטומטית את ה-workflow שב-
[`.github/workflows/deploy.yml`](./.github/workflows/deploy.yml): הוא מתקין
תלויות, בונה את האתר עם המשתנים שהגדרתם, ומפרסם ל-Pages. ניתן גם להריץ ידנית
מלשונית **Actions** → **Deploy to GitHub Pages** → **Run workflow**.

בסיום, הכתובת תופיע ב-**Settings → Pages**. דומיין מותאם אישית מגדירים מאוחר יותר
באותו מסך.

---

## בדיקה מהירה שהכל עובד
1. פתחו את כתובת ה-Pages.
2. מלאו את אשף הכרטיסים ולחצו "מעבר לתשלום".
3. ודאו שהופניתם לכתובת התשלום, ושב-Supabase → Table Editor → `ticket_clicks`
   נוספה שורה חדשה עם הפרטים.

אם לא נוספה שורה: בדפדפן פתחו Console (F12) — שגיאות מ-Supabase יודפסו שם.
הסיבות הנפוצות: משתנים שגויים, או ש-`schema.sql` לא הורץ.
